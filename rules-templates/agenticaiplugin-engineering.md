# Engineering Principles

## 1. Bidirectional Traceability

Every implementation references its requirement (STORY-XXX AC-Y).
Every requirement must have a corresponding implementation — prevents both forgetting requirements and building unrequested features.

```
// STORY-012 AC-1: Email validation per RFC 5322
if (!validateEmail(email)) {
    throw new InvalidEmailException();
}

// STORY-042 AC-3: Failed messages sent to DLQ after 3 retries
for (int retry = 0; retry < 3; retry++) { ... }
```

```
/**
 * Implements STORY-042: Dead Letter Queue Feature
 * - AC-1: Failed messages sent to DLQ after 3 retries
 * - AC-2: DLQ messages logged with error details
 * - AC-3: DLQ has separate topic
 */
public class DeadLetterQueueService { ... }
```

## 2. Encapsulation & API Surface

- Never widen API surface just for testability (no package-private/protected methods exposed only for tests)
- Test only through the public API
- If internal logic is complex enough to need dedicated testing → extract it into its own class with its own public API

## 3. Code Size Limits

| Scope | Preferred | Maximum |
|-------|-----------|---------|
| Method | <20 lines | 50 lines |
| Class | <200 lines | 500 lines |

Exceeding maximum → extract helper methods or split class.

## 4. Test Classification

"Test YOUR Code, Not THE Code"

| Code Type | Test? | Examples |
|-----------|-------|---------|
| Business Logic | YES | Services, domain logic, algorithms, validators |
| Custom Queries | YES | @Query("SELECT ...") |
| Infrastructure (custom) | PARTIAL | Retry/fallback logic |
| Framework Code | NO | Spring Boot config, controller routing, JPA mappings |
| Generated Code | NO | Lombok, MapStruct, QueryDSL |

When intentionally not testing, document with: `// No test: [reason]`

## 5. Dependency Management

Only add dependencies that are: (1) already in tech stack, (2) required for story AC, (3) explicitly approved by user.
